# GetSms — 设计文档

日期：2026-07-17
状态：已与用户确认，待评审

## 一句话

一个开源、自托管、单用户的服务：把你自己多台手机收到的短信全部汇总落盘，供你自己的 AI agent 通过 MCP、在给定关键词的前提下取用验证码。

## 背景与关键约束

移动系统厂商正在主动封堵「第三方读短信」这条路，这是本项目最大的现实约束（不是写代码）：

- **iOS**：第三方 App 无法读取短信（Apple 无此 API）。唯一消费级路径是用户自己配置的**快捷指令「消息」自动化** → `Get Contents of URL` POST。iOS 17+ 支持「立即运行」（无需确认点按），锁屏下可跑。已有商业 App 采用同款模式。已知坑：更新系统后自动化可能静默失效；「发件人」筛选**填不了短号**，需用「消息内容包含」绕过。
- **Android**：技术上可用 `RECEIVE_SMS` 广播接收器，但 **Android 17（2026-06）对非默认短信 App 延迟 3 小时才可见 OTP**；Google Play 与国内厂商（小米明确拒「读验证码」）均不批此用途。因此 v1 不自研 App。

**结论（设计取向）**：GetSms **不发布任何读短信的 App**，设备端复用现成自动化工具（v1：iOS 快捷指令；v2：Android SmsForwarder / Tasker）。GetSms 本体只是「服务端 + MCP Server」。

## 目标

- 单用户自托管，一份部署服务一个人。
- 支持该用户的多台手机 / 多个号码，可按设备区分。
- 存下所有进来的短信（持久化）。
- 对 AI agent 以 MCP 暴露「按关键词取验证码」能力。
- 开源，他人可 `docker compose up` 一键自部署。
- 无遥测、无对外连接，代码可审计。

## 非目标（YAGNI）

- 多用户 / SaaS / 账号体系。
- 短信池、号码分配、代收码等对外服务形态。
- 通用「查任意短信」工具（v1 只做取验证码接口）。
- 自研设备端 App。
- Android 教程（列入 v2）。
- 真正的端到端加密（与「持久化存所有短信」本就冲突，不做）。

## 架构与数据流

```
[某台手机]                       [服务端(单进程)]                [你的 agent]
收到验证码短信                                                    需要验证码
   │                                                               │
   │ 现成自动化(iOS快捷指令)                                        │ MCP: get_latest_code
   │ POST /ingest/{设备token}                                      │ (keyword 必填)
   ▼                                                               ▼
 ┌──────────────────┐  原样落盘+设备标签  ┌──────────────┐  查询  ┌──────────────┐
 │ Ingest HTTP 接口 │ ─────────────────▶ │ SQLite 持久库 │ ◀──── │  MCP Server  │
 │ /ingest/:token   │                    │ 全量短信存档   │       │              │
 └──────────────────┘                    └──────────────┘       └──────────────┘
```

进来的短信**原样落盘**（不在写入时判断是不是验证码），验证码的识别与过滤放在**查询时**做。

## 组件

### 1. Ingest 接口
- `POST /ingest/{token}`，body 为短信原文（及可选发件人、时间戳）。
- 按 URL 里的 `token` 在 config 的设备清单中匹配出**设备标签**；匹配不到则 401，不写入。
- 将短信原文、设备标签、接收时间**原样写入 SQLite**（不在此处抽码、不过滤）。

### 2. 存储（SQLite 持久化）
- 一张 `messages` 表：`id, device_label, body, sender?, received_at, ingested_at`。
- **持久落盘，默认不自动删除**——是你所有短信的长期存档。
- DB 文件权限收紧（仅属主可读写）；路径可配置。
- 抽象一层存储接口，便于日后换实现。

### 3. MCP Server
- 与 Ingest 同进程。
- **唯一工具 `get_latest_code({ keyword, device?, since? })`**：
  - **`keyword` 必填**（代表短信来源/标题，即国内短信开头 `【】` 里的签名，如 `淘宝`、`京东`、`12306`，或英文服务名）。**不传 keyword → 直接返回空，不做任何检索**。这是有意的安全约束：agent 必须知道自己要哪个来源的码。
  - 匹配条件：短信 body **包含 `keyword`**（大小写不敏感，匹配整条 body，天然覆盖 `【淘宝】` 这类签名）**且** body 命中「验证码指示词」表。
  - 「验证码指示词」表（可配置，默认）：中文 `验证码`；英文 `verification code` / `verification` / `code` / `OTP` / `one-time` / `passcode`。
  - `device` → 再按设备标签过滤。
  - **默认只看最近一段时间窗内的短信（默认 10 分钟，可配置），避免取到已作废的旧码。** `since` 显式给出时可覆盖此默认（放宽或收紧）。
  - 命中后从该条 body 抽取验证码（正则：4–8 位数字，容纳常见格式；抽不到则返回原文、码为空）。
  - 返回时间窗内最近一条匹配：验证码、原文、设备标签、时间。
- 认证：整份部署一个 MCP api key。

## 配置

一个 config 文件 / env：
- 设备清单：每台 `{ 标签, ingest token }`。
- 一个 MCP api key。
- SQLite 文件路径。
- 验证码指示词表（可覆盖默认）。
- 取码默认时间窗（默认 10 分钟）。
- 监听端口等。

开源交付物：`docker-compose.yml`（一键起）、`.env.example`、iOS 快捷指令图文配置教程、`LICENSE`、`README`。

## 安全模型

- **单用户自托管**，无外部账号，攻击面小。
- 每设备独立 ingest token：泄露只波及该设备，撤换单条即可。
- **诚实交底（重要）**：服务端会**持久保存你所有短信的明文存档**，不是短命中转。这比只存验证码敏感得多——一旦服务器被入侵，泄露的是你的全部短信历史。缓解：DB 文件权限收紧、部署在你信任的私有机器、全程 TLS（由部署方自行上反代 / 证书）。README 需明确警示这一点。
- 取码接口的 `keyword` 必填是纵深防御：即便 MCP api key 泄露，攻击者也得逐个来源猜关键词，拿不到「所有验证码」的全集。
- 无遥测、无主动外连，行为可从代码审计。

## 错误处理

- Ingest token 无效 → 401，不写入。
- `get_latest_code` 未给 `keyword` → 返回空结果并说明「需提供关键词」。
- 有 `keyword` 但时间窗内无匹配短信 → 返回空结果（非报错），说明「暂无匹配（可能已超出时间窗或尚未收到）」。
- 命中短信但抽不到数字码 → 返回原文、码字段为空，交由 agent 判断。
- MCP api key 无效 → 拒绝。

## 测试

- Ingest：有效/无效 token、原文原样落盘、字段完整。
- 存储：写入/查询、按设备与时间过滤、DB 文件权限。
- 取码工具：
  - 无 keyword → 空。
  - keyword 命中 `【签名】` + 含「验证码」→ 返回码。
  - keyword 命中但非验证码短信（不含指示词）→ 不返回。
  - 中英文验证码短信各测；数字码抽取对多种格式。
  - 时间窗：窗内的返回、窗外的旧码不返回；`since` 覆盖默认窗。
  - `device` 过滤分支；鉴权。
- 端到端：模拟 POST 一条 `【淘宝】...验证码 123456...` → agent 带 keyword=淘宝 取到 123456。
