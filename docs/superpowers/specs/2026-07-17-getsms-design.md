# GetSms — 设计文档

日期：2026-07-17
状态：已与用户确认，待评审

## 一句话

一个开源、自托管、单用户的服务：把你自己多台手机收到的短信验证码汇总到一处，供你自己的 AI agent 通过 MCP 取用。

## 背景与关键约束

移动系统厂商正在主动封堵「第三方读短信」这条路，这是本项目最大的现实约束（不是写代码）：

- **iOS**：第三方 App 无法读取短信（Apple 无此 API）。唯一消费级路径是用户自己配置的**快捷指令「消息」自动化** → `Get Contents of URL` POST。iOS 17+ 支持「立即运行」（无需确认点按），锁屏下可跑。已有商业 App 采用同款模式。已知坑：更新系统后自动化可能静默失效；「发件人」筛选**填不了短号**，需用「消息内容包含」绕过。
- **Android**：技术上可用 `RECEIVE_SMS` 广播接收器，但 **Android 17（2026-06）对非默认短信 App 延迟 3 小时才可见 OTP**；Google Play 与国内厂商（小米明确拒「读验证码」）均不批此用途。因此 v1 不自研 App。

**结论（设计取向）**：GetSms **不发布任何读短信的 App**，设备端复用现成自动化工具（v1：iOS 快捷指令；v2：Android SmsForwarder / Tasker）。GetSms 本体只是「服务端 + MCP Server」。

## 目标

- 单用户自托管，一份部署服务一个人。
- 支持该用户的多台手机 / 多个号码，可按设备区分。
- 对 AI agent 以 MCP 暴露「取最新验证码」能力。
- 开源，他人可 `docker compose up` 一键自部署。
- 无遥测、无对外连接，代码可审计。

## 非目标（YAGNI）

- 多用户 / SaaS / 账号体系。
- 短信池、号码分配、代收码等对外服务形态。
- 数据库持久化短信。
- 自研设备端 App。
- Android 教程（列入 v2）。
- 真正的端到端加密（列入 v2+，需设备端换成能做非对称加密的自研 App）。

## 架构与数据流

```
[某台手机]                       [服务端(单进程)]                [你的 agent]
收到验证码短信                                                    需要验证码
   │                                                               │
   │ 现成自动化(iOS快捷指令)                                        │ MCP: get_latest_code
   │ POST /ingest/{设备token}                                      │
   ▼                                                               ▼
 ┌──────────────────┐  抽码+设备标签  ┌───────────────┐  读取  ┌──────────────┐
 │ Ingest HTTP 接口 │ ─────────────▶ │ 内存临时存储    │ ◀──── │  MCP Server  │
 │ /ingest/:token   │                │ 按设备keyed,TTL │       │              │
 └──────────────────┘                └───────────────┘       └──────────────┘
```

一条验证码在服务端只活几分钟：POST 进来 → 抽码 → 写入带 TTL 的内存存储 → agent 取走或超时消失。**不落盘明文短信。**

## 组件

### 1. Ingest 接口
- `POST /ingest/{token}`，body 为短信原文（及可选发件人、时间戳）。
- 按 URL 里的 `token` 在 config 的设备清单中匹配出**设备标签**；匹配不到则 401。
- 用正则从原文抽取验证码（4–8 位数字，容纳常见格式）；抽不到也保留原文。
- 连同设备标签、原文、时间戳写入临时存储。

### 2. 临时存储
- 键：设备标签；值：该设备最近的若干条验证码记录（含码、原文、时间）。
- 每条带 TTL，默认约 5 分钟，定时清理过期项。
- v1 用进程内存实现；对外以一个存储接口封装，便于 v2 换 Redis（多实例时）。

### 3. MCP Server
- 与 Ingest 同进程。
- 工具 `get_latest_code({ device?, contains?, since? })`：
  - 不传 `device` → 返回全局最新一条；传了 → 按设备标签筛。
  - `contains` → 按原文关键词过滤（对应快捷指令短号绕法）。
  - `since` → 只要该时间点之后的。
  - 返回：验证码、原文、设备标签、时间。
- 认证：整份部署一个 MCP api key。

## 配置

一个 config 文件 / env：
- 设备清单：每台 `{ 标签, ingest token }`。
- 一个 MCP api key。
- TTL 时长（默认 5 分钟）。
- 监听端口等。

开源交付物：`docker-compose.yml`（一键起）、`.env.example`、iOS 快捷指令图文配置教程、`LICENSE`、`README`。

## 安全模型

- **单用户自托管**，无外部账号，攻击面小。
- 每设备独立 ingest token：泄露只波及该设备，撤换单条即可。
- 短信仅在内存短暂停留，不持久化明文。
- 全程依赖 TLS（由部署方自行上反代 / 证书）。
- 无遥测、无主动外连，行为可从代码审计。
- **诚实交底**：这是「短命中转」而非 E2E——取码那一刻服务端必然看得到明文。真 E2E 需设备端能对 agent 公钥做非对称加密，属 v2+。

## 错误处理

- Ingest token 无效 → 401，不写入。
- 抽不到验证码 → 仍保留原文，`get_latest_code` 返回码为空但带原文，交由 agent 判断。
- 存储中无匹配记录 → MCP 工具返回空结果（非报错），并说明是「暂无 / 已过期」。
- MCP api key 无效 → 拒绝。

## 测试

- Ingest：有效/无效 token、抽码正则对多种验证码格式、TTL 过期清理。
- 存储：按设备读写、TTL 到期后不可读、每设备保留条数上限。
- MCP：`device` / `contains` / `since` 各筛选分支、空结果、鉴权。
- 端到端：模拟一次 POST → MCP 取到该码。
